# 系统架构设计

（基于本项目的学术化表述，可用于毕业论文“系统架构设计”或“总体设计”章节）

---

## 一、总体架构与设计目标

本系统旨在在**本地、离线**环境下，依托小参数规模因果语言模型（如 Llama3.2 1B、Qwen2.5 1.5B）实现邮件的智能分类、简短回复生成与正文润色，并支持多邮箱账号的添加、切换与持久化。为兼顾可维护性、可扩展性与资源占用，系统采用**分层、模块化**的架构设计：将邮件接入与解析、语言模型推理、业务逻辑与用户界面分离，各层通过**统一数据结构**与**明确接口契约**进行协作，避免模块间紧耦合，并便于后续替换或扩展某一层实现（如更换模型后端、增加新邮箱协议支持等）。

---

## 二、分层架构与模块划分

系统自下而上划分为**四层**：配置与数据层、邮件接入与解析层、本地大模型推理与业务逻辑层、表示层。各层职责与主要模块如下。

### 2.1 配置与数据层

该层提供**邮箱服务器配置**与**账号持久化**，不直接参与邮件收发或模型推理，仅为上层提供静态配置与持久化数据。

- **邮箱服务器配置（config / email_servers）**：根据用户邮箱地址的域名（如 @163.com、@qq.com）查询预定义的 IMAP/SMTP 服务器地址、端口及是否使用 SSL，支持 QQ、网易、Gmail、Outlook、阿里云等常见服务商，使用户无需手工填写服务器信息即可添加账号。
- **账号存储（core / account_store）**：在内存中维护多账号列表及当前选中下标，提供账号的增删改查与当前账号切换；将账号信息（邮箱、密码或授权码、IMAP/SMTP 配置）以 JSON 格式持久化至用户目录下的固定路径，实现“账号记忆”与跨会话恢复。该模块使用可重入锁（RLock）保证多线程访问下的数据一致性。

### 2.2 邮件接入与解析层

该层负责与邮件服务器通信，拉取原始邮件数据，并将其解析为应用内统一的**结构化表示**，供分类、回复与界面展示使用。

- **邮件客户端（core / email_client）**：封装单账号维度的 IMAP 收信与 SMTP 发信。基于标准库 imaplib、smtplib 及 ssl 模块建立安全连接；在 IMAP 登录后可选发送 IMAP ID 命令以通过部分服务商的安全策略；提供 `fetch_recent(n, folder)` 拉取指定文件夹最近 n 封邮件的 RFC822 原始数据，以及 `send_mail(to, subject, body, attachments)` 发送纯文本或带附件的邮件。连接建立与 SELECT/SEARCH 失败时，将服务器返回信息解码后纳入异常消息，便于诊断。
- **邮件解析（utils / email_parser）**：将单封邮件的原始字节流解析为 `MailItem` 数据结构，包括发件人、主题、日期、纯文本正文、HTML 正文、附件列表与正文内链接列表；对 MIME 头进行解码、对 HTML 做简易转纯文本处理，并提取附件元信息与 payload，为上层提供统一的“单封邮件”抽象。

### 2.3 本地大模型推理与业务逻辑层

该层是系统的**核心计算层**：对外提供统一的本地语言模型推理能力，并在此基础上实现邮件分类、回复生成与正文润色等业务接口；同时引入缓存以降低重复推理开销。

- **本地大模型封装（core / local_llm）**：采用**单例模式**与**双后端抽象**（Transformers 与 Ollama），对外提供统一的 `generate(prompt, temperature, max_tokens, use_cache)` 接口；内部根据配置或环境变量选择从本地目录加载 Transformers 模型或通过 HTTP 调用 Ollama 服务，并可选地将生成结果写入 LRU 风格缓存。该设计使上层业务不关心具体推理实现，仅通过 prompt 与参数控制行为。
- **业务接口**：在 LocalLLM 之上实现三类业务方法——**垃圾邮件二分类**（classify_spam）、**自动回复生成**（generate_reply）、**邮件正文润色**（polish_email）——分别对应不同的提示词设计、生成参数与后处理流水线；分类结果可被表示层缓存并用于列表筛选与标色。
- **缓存（utils / cache）**：提供线程安全的、容量受限的键值缓存（SimpleCache），用于缓存分类与生成结果，避免对相同或近似输入重复调用模型，在保证一致性的同时降低延迟与资源消耗。

### 2.4 表示层

该层负责**用户交互**与**异步调度**：通过图形界面展示邮箱列表、邮件正文、分类结果与回复编辑区，并将耗时操作（拉取邮件、分类、生成回复、润色、发送）委托给后台工作线程执行，通过信号将结果或异常回传主线程，避免界面阻塞并满足 GUI 线程安全要求。

- **主窗口（ui / main_window）**：基于 PyQt5 构建，包含邮箱与模型选择、邮件列表与筛选、正文与附件/链接展示、回复编辑区、分类与回复/润色/发送等操作按钮；持有对 AccountStore、EmailClient、LocalLLM 的引用，在用户触发操作时调用相应模块并更新界面状态。
- **工作线程（WorkerThread）**：继承 QThread，将拉取邮件、批量分类、生成回复、润色、发送等可阻塞操作封装为在子线程中执行的函数，通过 `finished_signal` 与 `error_signal` 将结果或异常传递回主线程，由主窗口更新 UI 或弹出错误提示。该设计保证“界面响应”与“耗时计算”分离，符合桌面应用的最佳实践。
- **添加账号对话框（ui / add_account_dialog）**：提供邮箱地址与密码（或授权码）输入，并根据域名从配置层获取 IMAP/SMTP 信息；确认后构造 AccountItem 并交由 AccountStore 持久化，主窗口据此刷新账号列表并可选触发首次拉取。

## 三、数据流与调用关系

- **启动与初始化**：程序入口（main.py）将项目根目录加入模块搜索路径，创建 QApplication 与 MainWindow；主窗口初始化时加载 AccountStore 中的持久化账号、扫描本地 models 目录及环境变量以填充“当前模型”下拉框，并可根据当前选中账号在用户首次点击“刷新邮件”时按需创建 EmailClient 与建立 IMAP 连接。
- **拉取与展示邮件**：用户选择账号并点击刷新后，主窗口在工作线程中调用 EmailClient.fetch_recent()，获取 MailItem 列表；解析层已在 fetch 过程中通过 email_parser 完成。主窗口将列表渲染至左侧邮件列表，并在用户点击某封邮件时将其正文、附件、链接与发送时间展示于右侧，同时可加载已缓存的分类结果并更新“分类”标签。
- **分类、回复与润色**：用户点击“智能分类”“生成回复”或“润色”时，主窗口在工作线程中先确保 LocalLLM 已加载（若未加载则根据环境选择 Transformers 或 Ollama），再调用 classify_spam、generate_reply 或 polish_email，并将返回结果通过信号传回主线程，写回界面（如分类标签、回复编辑框）。若某步抛出异常，则由 error_signal 触发错误弹窗。
- **发送邮件**：用户编辑回复并可选添加本地附件后点击“发送”，主窗口在工作线程中调用 EmailClient.send_mail()，将收件人、主题、正文及附件列表传入；发送成功后清空回复区与待发送附件列表并提示用户。

上述流程中，**MailItem** 与 **AccountItem** 作为跨层传递的核心数据结构，保证从解析到展示、从配置到发信的数据语义一致。

## 四、关键设计决策

- **单例 + 双后端的本地 LLM**：全局仅维护一个 LocalLLM 实例，避免多份模型占用内存；通过 Transformers 与 Ollama 两种后端抽象，支持“本地权重文件”与“本地推理服务”两种部署方式，用户可通过界面或环境变量切换，无需修改业务代码。
- **延迟加载与按需连接**：模型在首次执行分类或生成时才加载；EmailClient 在首次拉取或发送时才建立 IMAP/SMTP 连接，并尽量复用已有连接（如通过 noop 检测存活），以缩短启动时间并减少无效连接。
- **异步执行与信号驱动**：所有可能阻塞的 I/O 与推理均在 QThread 中执行，结果通过 Qt 信号槽回传主线程更新 UI，既保证界面流畅又避免跨线程直接操作控件带来的稳定性问题。
- **配置与模型路径外置**：邮箱服务器配置、账号存储路径、模型目录（models 及 LLAMA_MODEL_PATH）、Ollama 地址与模型名等均通过配置文件或环境变量指定，便于在不同环境部署或打包分发，而无需改代码。

---

## 五、小结

本系统采用**分层、模块化**的架构设计，将配置与数据、邮件接入与解析、本地大模型推理与业务逻辑、表示层清晰分离，通过统一数据结构（MailItem、AccountItem）与明确接口完成层间协作；通过单例双后端 LLM、延迟加载、异步工作线程与配置外置等设计，在保证功能完整与可维护性的前提下，兼顾启动速度、运行期资源占用与多环境适配。上述系统架构设计可作为毕业论文“系统架构设计”或“总体设计”章节的撰写依据，并可据此绘制分层架构图或模块依赖图以配合说明。
