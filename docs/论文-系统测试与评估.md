# 系统测试与评估（论文正文，无小标题）

本节从测试环境、功能验证、性能与效果观察及评估结论四个方面对系统进行测试与评估，以验证各模块的正确性、稳定性与实际可用性。

测试在 Windows 环境下进行，采用 Python 3.10 及以上解释器，依赖 PyQt5、Transformers、sentencepiece 等库；本地推理支持 CPU 与 CUDA 两种设备，Ollama 后端需在本地部署并开放默认端口。测试用邮箱覆盖常见 IMAP/SMTP 服务商，邮件样本包含个人往来、工作通知、营销推广及带广告标记的群发邮件，以检验分类与回复在不同类型邮件上的表现。功能测试围绕端到端流程展开：账号添加与持久化、多账号切换、邮箱服务器自动配置与手工配置、IMAP 拉取与 SELECT 失败时的错误提示、邮件解析对纯文本与 HTML 正文及附件的提取、主题与正文内链接的识别与展示、分类规则与模型判别的协同及缓存命中、回复生成的提示约束与后处理去重与截断、润色三种风格的切换与长度控制、带附件的发送及界面在工作线程下的响应与异常提示。上述用例在 Transformers 与 Ollama 两种后端下分别执行，确认双后端接口一致性与单例切换后重新加载的正确性。

性能与效果方面，在典型硬件上对 7B 级模型进行观察：分类任务因规则预筛与缓存，对已判定或重复邮件几乎无额外延迟，对需调用模型的样本单次推理在数秒量级；回复与润色受生成长度与温度影响，时延在数秒至十余秒之间，界面通过异步线程未出现卡顿。分类效果上，对含明显广告标记或营销关键词的邮件能稳定判为垃圾，对简短私人邮件在规则与提示词约束下多判为正常，对边界样本依赖模型能力，存在少量误判可能；回复与润色在约束下能产出简短、可直接使用的文本，偶有风格或长度需用户微调。评估结论如下：系统在既定环境下能够完成邮件拉取、分类、回复生成与润色及发送的完整流程，功能符合设计目标，双后端与缓存机制有利于降低推理次数与等待时间；分类与生成质量受本地模型规模与提示设计限制，未与外部 API 或大规模标注数据做对比实验，评估以功能正确性与实际使用观察为主。后续可从引入标注数据集进行分类准确率与 F1 的定量评估、对回复与润色结果进行人工打分或与参考回复的相似度分析等方面深化评估工作。
